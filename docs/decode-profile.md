# 압축 해제 경로 프로파일 (2026-07-20)

`docs/archive-roadmap.md`가 기록해 온 두 가지 추정 —

- "디코드가 전체의 10–20%라 멤버 파티션 병렬은 이득이 작다" (roadmap 3번)
- "t≥4는 디코드 직렬 플로어" (roadmap 스레드 스케일링 표)

— 를 계측으로 검증한 결과 **둘 다 사실과 다릅니다.** 이 문서는 그 측정 기록입니다.

## 측정 환경

| | 개발 머신 | 정본 머신 |
|---|---|---|
| CPU | Apple M1, 8코어 | Xeon x86_64, 32코어 |
| OS | macOS 15.7.7 (arm64) | Rocky Linux 8.10 (Docker, Ubuntu 22.04 컨테이너) |
| RAM | 16 GB | 125 GB |
| 빌드 | RelWithDebInfo (프로파일용) / Release | Release |
| 계측 | `sample` 1ms 간격 | 벽시계 A/B |

픽스처는 `docloom-corpus/hwp` 177문서(641MB)를 동일 내용으로 store(method 0)와
deflate(method 8)로 각각 포장한 쌍. 정본 머신의 `large_docs_xl.zip`이 이미 deflate 177개
동일 구성이라 그대로 사용하고, store 쌍만 새로 생성했습니다. 모든 측정은 페이지캐시
워밍 후 수행했습니다.

**store/deflate 쌍 비교가 이 문서의 핵심 기법입니다.** 내용·엔트리 수·파서 경로가 완전히
동일하고 압축 방식만 다르므로, 두 실행의 차이는 정의상 inflate 비용입니다.

## 1. deflate zip — inflate가 지배적

### 정본 머신 (Linux x86_64, Release)

| 워크로드 | t=1 | t=2 | t=4 | t=8 | t=16 |
|---|---|---|---|---|---|
| store (641MB, 177문서) | 0.486s | 0.454s | 0.467s | 0.455s | 0.444s |
| deflate (동일 내용) | 2.161s | 2.177s | 2.005s | 1.994s | 1.927s |

**inflate 비용 = 2.161 − 0.486 = 1.675s → deflate 실행의 77%.**

### 개발 머신 (macOS arm64, 프로파일러)

`sample` 8488 샘플 중:

| 구간 | 샘플 | 비중 |
|---|---|---|
| zlib inflate (`read_entry_streamed`의 `zip_reader.cpp:252` 서브트리) | 7060 | **83%** |
| read syscall | 628 | 7.4% |
| 문서 파싱 | 414 | 4.9% |

콜그래프로 귀속을 확인했습니다. libz의 미노출 심볼(`inflate_fast` 등) 전량이
`ZipReader::read_entry_streamed`의 inflate 분기 아래에 있습니다.

**두 플랫폼이 독립적인 방법으로 77% / 83%를 가리킵니다. roadmap의 "10–20%"는 기각됩니다.**

### 왜 이렇게 큰가

hwp·pdf·docx·xlsx는 모두 내부적으로 이미 압축된 포맷입니다. 이들을 zip으로 묶으면
압축률은 2%에 불과한데(641MB → 659MB, 오히려 store 672MB와 큰 차이 없음) **inflate 전량
비용은 그대로** 냅니다. zip 기본값이 deflate이므로 이것이 실사용 대표 케이스입니다.

## 2. 스레드가 전혀 듣지 않는다

roadmap은 "t≥4 평탄화"를 기록했지만, 실제로는 **t=1부터 평탄**합니다.

- 정본 store: 0.486 → 0.444s (t=1→16). 32코어를 줘도 **8%**.
- 정본 deflate: 2.161 → 1.927s. **11%**.
- 개발 deflate: 2.21 → 2.11s. **5%**.

원인은 명확합니다. `member_pipeline.h`의 설계대로 **디코드·순회·캡 회계는 호출 스레드에
남고 문서 변환만 워커로 나갑니다.** 그런데 위 측정대로 이 워크로드들은 변환이 아니라
디코드/워크가 지배하므로, 병렬화된 부분이 애초에 작습니다.

roadmap이 이를 "디코드 직렬 플로어"라 부른 것은 deflate에 대해서는 맞는 진단이지만,
**store 워크로드에도 같은 라벨을 붙인 것은 틀렸습니다** — store는 inflate가 아예 없습니다.
store의 플로어는 별개 원인이고, 아래 4번이 그 정체입니다.

## 3. mmap은 플랫폼에 따라 정반대 (중요)

2.5–3.1GB 파일을 워밍된 페이지캐시에서 읽는 마이크로벤치(`iobench.c`, 각 2회):

| 방식 | macOS arm64 (M1) | Linux x86_64 (Rocky) |
|---|---|---|
| `fread` 64KB 청크 (현행) | 0.83–0.99s (2.7–3.3 GB/s) | 0.56–0.61s (5.4–5.9 GB/s) |
| `mmap` + memcpy | 5.81–9.08s (0.30–0.46 GB/s) | 0.47s (7.0 GB/s) |
| **`mmap` 제로카피(touch만)** | 6.55–8.17s (0.33–0.37 GB/s) | **0.17–0.22s (15–20 GB/s)** |
| `mmap` + 프리폴트(`MADV_WILLNEED`/`MAP_POPULATE`) | 8.92–10.82s (더 악화) | 0.44–0.50s |
| `read`(2) → 대형 단일 버퍼 | 1.56–1.89s (1.4–1.7 GB/s) | 2.26–2.40s (1.4–1.5 GB/s) |

- **Linux: mmap 제로카피가 fread 대비 3배 빠름.** 운영 타깃에서 큰 이득.
- **macOS: mmap이 fread 대비 8배 느림.** 페이지 폴트 비용이 압도적이고 프리폴트로도 개선되지
  않습니다(오히려 악화).

따라서 mmap은 **Linux 한정 옵트인**으로만 도입할 수 있습니다. roadmap이 기록한 보류 사유
(maxRSS 지표 왜곡)와는 무관한, 훨씬 실질적인 제약입니다.

## 4. store 워크로드의 진짜 병목은 익명 페이지 폴트

정본 머신 store 플로어 0.444s는 641MB 기준 **1.46 GB/s**입니다. 그런데 같은 머신의 `fread`는
5.4 GB/s입니다 — **파일 읽기는 병목이 아닙니다.**

1.46 GB/s와 정확히 일치하는 것은 위 표의 `read`→대형 단일 버퍼 케이스(1.4–1.5 GB/s)입니다.
이 케이스가 느린 이유는 파일 I/O가 아니라 **갓 할당된 익명 메모리에 처음 쓸 때 발생하는
페이지 폴트**입니다.

JDoc이 멤버마다 하는 일이 정확히 이것입니다 (`archive_walker.cpp:378-380`):

```cpp
std::vector<char> data;
data.reserve(std::min<uint64_t>(e.uncompressed_size, opts.archive.max_member_bytes));
```

멤버마다 새 버퍼를 할당하고, 다 쓰면 해제합니다. 할당자는 큰 블록을 OS에 반납하고
(개발 머신 프로파일에서 `madvise` 63샘플 = 7%로 관측됨) 다음 멤버에서 다시 폴트합니다.

**이건 버퍼를 재사용하면 플랫폼 무관하게 제거할 수 있습니다.** mmap보다 싸고 이식성이 있어
우선순위가 높습니다.

## 5. PDF는 flate 병목이 아니다

`bench_convert dir ~/docloom-corpus/pdf`, 7422 샘플:

| 구간 | 샘플 | 비중 |
|---|---|---|
| `parse_content_stream` (+람다) | 1103 | 14.9% |
| `render_page_composite` (+스캔라인 람다) | 1122 | 15.1% |
| `memcmp` / `strlen` | 678 | 9.1% |
| **zlib (`decode_flate` 서브트리)** | ~620 | **8.4%** |
| allocator `madvise` | 232 | 3.1% |

PDF가 4.6MB/s로 전 포맷 중 최악인 것은 사실이지만 **원인은 압축 해제가 아니라 콘텐츠 스트림
파싱과 래스터라이즈**입니다. PDF flate 특화(z_stream 재사용 등)는 상한이 8%이므로 기각하고,
PDF는 압축 해제와 별개 과제로 분리합니다.

## 결론 — 우선순위

작업 범위는 **단일 코어 성능**으로 잡았습니다. 멀티코어 전용 항목은 현재 `--threads`가
이 워크로드들에서 8–11%밖에 못 내는 것과 별개로, 단일 코어 이득이 0이라 제외했습니다.

| 상태 | 항목 | 근거 |
|---|---|---|
| ✅ 적용 | mmap 제로카피 (**Linux 기본 ON**) | store −82%, 피크 RSS −82% (부록 E) |
| ✅ 적용 | 멤버 버퍼 재사용 | store −16%, 피크 RSS −9~17% (부록 D) |
| ✅ 적용 | 기본 빌드 타입 Release | `-O0` 배포 사고 방지 (부록 B) |
| ❌ 기각 | libdeflate | 상한 2.6%, 실측 +10% 회귀 (부록 C) |
| ❌ 기각 | LTO | 측정 결과 순손실 (부록 A) |
| ❌ 기각 | PDF flate 특화 | 상한 8%, PDF는 파서 병목 (5번) |
| 범위 외 | 멤버 디코드 병렬화 | 단일 코어 이득 0, 캡 회계 결정성 재설계 필요 |

**이 문서의 교훈**: 프로파일이 지목한 최대 항목(inflate 77–83%)조차 코덱을 바꾸는 것으로는
회수되지 않았습니다. 이 경로는 연산이 아니라 **메모리 대역폭**에 막혀 있고, 그래서 남은
후보 두 개(버퍼 재사용, mmap)도 모두 "메모리 트래픽을 줄이는" 방향입니다. 착수 전 같은
방식으로 상한을 먼저 재는 것을 권합니다.

## 재현 방법

```bash
# 프로파일 빌드
cmake -S . -B build-prof -DCMAKE_BUILD_TYPE=RelWithDebInfo \
      -DCMAKE_CXX_FLAGS="-fno-omit-frame-pointer"
cmake --build build-prof -j --target bench_convert

# store/deflate 쌍 생성 (동일 내용)
cd ~/docloom-corpus && zip -0 -q -r /tmp/store.zip hwp && zip -6 -q -r /tmp/deflate.zip hwp

# 워밍 후 스레드 스윕
cat /tmp/store.zip > /dev/null
for t in 1 2 4 8 16; do ./build-prof/bench_convert archive /tmp/store.zip $t; done

# macOS 프로파일링 (프로세스가 샘플링 창보다 오래 살아야 함)
./build-prof/bench_convert archive /tmp/deflate.zip 1 & sample $! 12 1 -mayDie -f prof.txt

# I/O 마이크로벤치 (mmap 플랫폼 차이)
cc -O2 -w -o iobench test/bench/iobench.c && ./iobench /tmp/store.zip 64
```

## 부록 A. LTO A/B (기각)

계획 단계에서 "크로스-TU 인라이닝 여지가 크므로 3–8% 이득"으로 예상했으나 **측정 결과
순손실**이라 `JDOC_LTO` 기본값을 OFF로 두었습니다.

정본 머신(Rocky 8.10 x86_64, Release, 5회 중앙값), Clang ThinLTO:

| 워크로드 | no-LTO | LTO | 차이 |
|---|---|---|---|
| store zip 641MB | 0.437s | 0.457s | +4.6% (느려짐) |
| deflate zip 641MB | 1.894s | 1.861s | −1.7% (노이즈 수준) |
| dir ext-samples (43문서) | 2.326s | 2.457s | +5.6% (느려짐) |

출력은 양쪽 동일(`--hash` 게이트 통과). deflate 워크로드에서 이득이 없는 것은 예상 가능한
결과입니다 — 그 워크로드의 83%가 외부 `libz`의 `inflate`라 LTO 대상이 아닙니다.

**개발 머신(macOS)에서는 판정 불가였습니다.** 같은 빌드의 5회 측정이 0.51–2.23s로 흩어져
4–6% 차이를 분해할 수 없었습니다. 성능 A/B는 조용한 정본 머신에서 수행해야 합니다.

## 부록 B. 기본 빌드 타입

LTO와 함께 검토하다 발견한 별개 문제입니다. 단일 구성 제너레이터에서 `CMAKE_BUILD_TYPE`이
비어 있으면 CMake는 최적화 플래그를 **하나도** 넣지 않습니다. 즉 `cmake -S . -B build`만 한
사용자는 `-O0` 바이너리를 얻고, 그 사실이 빌드 로그에 드러나지 않습니다.

README와 벤치 스크립트는 모두 `-DCMAKE_BUILD_TYPE=Release`를 명시하고 있어 측정에는 영향이
없었지만, 라이브러리를 그냥 빌드해 쓰는 경로에서는 사고가 됩니다. `CMakeLists.txt`에서
비어 있을 때 Release로 채우도록 했습니다.

## 부록 C. libdeflate A/B (기각)

inflate가 77–83%라는 위 측정 때문에 **가장 유망한 항목**으로 꼽았지만, 구현해서 재보니
**순손실**이라 되돌렸습니다.

### 코덱 단독으로는 실제로 빠르다

정본 머신(Xeon Gold 5315Y, zlib 1.2.11), 32MB 버퍼를 5회 반복 디코드:

| 입력 | 압축비 | zlib | libdeflate 1.10 | libdeflate 1.25 |
|---|---|---|---|---|
| 랜덤(비압축성) | 1.00× | 3880 MB/s | 5569 MB/s (1.44×) | 5086 MB/s (1.37×) |
| 영문 텍스트 | 7.75× | 525 MB/s | 1270 MB/s (2.42×) | 1130 MB/s (2.52×) |
| 실제 hwp 멤버 | 1.14× | 275 MB/s | 379 MB/s (1.37×) | 338 MB/s (1.23×) |

### 그런데 파이프라인에서는 사라진다

`large_docs_xl.zip`(deflate 641MB, 177문서), 5회 중앙값:

| 구성 | 시간 | 대비 |
|---|---|---|
| zlib 스트리밍 (현행) | 1.882s | — |
| libdeflate one-shot + sink 복사 | 2.071s | **+10.0%** |
| libdeflate one-shot, sink 복사 제거 (상한 측정용) | 1.833s | −2.6% |

세 번째 줄은 출력을 버리는 계측 전용 빌드입니다. **중간 복사를 완전히 없애는 이상적인
통합조차 상한이 2.6%**라는 뜻이므로, 더 깊은 리팩터로도 회수할 수 없습니다.

### 왜 코덱 이득이 사라지는가

마이크로벤치는 2MB 버퍼를 캐시에 올려두고 반복 디코드하므로 **연산 바운드**입니다. 실제
워크로드는 641MB를 페이지캐시에서 한 번 흘려보내며 디코드하고 다시 멤버 버퍼로 복사하므로
**메모리 대역폭 바운드**입니다. 대역폭에 막힌 구간에서는 디코더의 연산 우위가 메모리 스톨
뒤에 가려집니다.

여기에 one-shot 특유의 비용이 더해집니다. 스트리밍은 64KB 중간 버퍼가 L1/L2에 상주하지만,
one-shot은 멤버 크기(수 MB) 중간 버퍼를 DRAM에 쓰고 다시 읽어 복사합니다 — 멤버당 DRAM
왕복이 두 번 늘어납니다.

### 구현 중 발견한 함정 (다시 시도한다면)

- `std::vector::resize()`는 **늘어난 영역을 0으로 채웁니다.** 멤버마다 크기가 다른 스크래치
  버퍼에 쓰면 멤버당 memset이 추가됩니다. 첫 측정에서 이득이 6%로 보인 원인이었고,
  0 채우기 없는 grow-only 버퍼로 바꾸자 22%까지 올라갔습니다(이후 정본 머신 재측정에서
  전체가 회귀로 판명).
- libdeflate에는 스트리밍 API가 없으므로 캡 동등성을 지키려면 **캡을 넘길 수 있는 멤버에는
  아예 one-shot을 쓰지 않는** 설계가 필요합니다(`uncompressed_size <= 유효캡`일 때만 진입).
  그래야 부분 출력과 에러 문구가 스트리밍 경로에서 그대로 재생됩니다. 이 방식으로 5개
  픽스처 + 코퍼스 521파일 전부 바이트 동일을 확인했으므로, 접근 자체는 유효했습니다 —
  기각 사유는 순전히 성능입니다.

### 남은 여지

`large_text_xl.zip`처럼 압축비가 높은 입력에서는 코덱 이득이 2.5배로 가장 크지만, 이
코퍼스의 실사용 입력(hwp/pdf/docx 등 이미 압축된 문서)은 압축비가 1.0~1.14라 해당되지
않습니다. **텍스트·로그 위주 아카이브가 주 대상이 된다면 재검토할 가치가 있습니다.**

## 부록 D. 멤버 버퍼 재사용 (적용)

4번에서 지목한 익명 페이지 폴트를 `WalkBudget::buffers`(`BufferPool`)로 제거했습니다.
멤버마다 새 `std::vector<char>`를 할당하는 대신 풀에서 빌려 쓰고 스코프 종료 시 반납하므로,
다음 멤버는 이미 폴트된 페이지에 씁니다.

중첩 아카이브가 재귀하는 동안 바깥 워크도 버퍼를 들고 있으므로, 단일 스크래치가 아니라
**빌려주는 풀**이어야 합니다(안쪽이 바깥쪽 버퍼를 `clear()`하면 손상). 풀은 재귀 깊이만큼만
자라고, 한 번 커진 용량이 워크 내내 붙들리지 않도록 반납 시 64MiB / 8개 상한을 둡니다.
병렬 경로는 워커가 버퍼를 가져가 빈 버퍼가 반납되므로 기존대로 멤버마다 할당합니다.

정본 머신, **교차 실행** A/B 각 9회 중앙값 (같은 조건에서 순차 실행하면 머신 드리프트가
한쪽에만 실려 5~10%씩 흔들렸습니다):

| 픽스처 | before | after | 시간 | 피크 RSS |
|---|---|---|---|---|
| store 641MB (177문서) | 0.535s | 0.449s | **−16.1%** | 166.9 → 151.4 MB (−9.3%) |
| deflate 641MB (177문서) | 1.907s | 1.834s | −3.8% | — |
| deflate 3.3GB (885문서) | 9.472s | 9.306s | −1.8% | 183.6 → 152.5 MB (**−17.0%**) |
| 5중 중첩 258MB | 0.659s | 0.667s | +1.2% (노이즈) | 218.1 → 217.8 MB |

deflate 워크로드에서 이득이 작은 것은 예상대로입니다 — 그쪽은 시간의 77%가 inflate라
버퍼 할당 비중이 작습니다. store 워크로드는 정반대라 이득이 큽니다.

**출력 동일성**: 10개 픽스처 × t=1/t=8 + 코퍼스 디렉터리 전부 바이트 동일. 중첩 아카이브
(`nested_xl`, `nested`, `diag_nested`)를 포함해 검증했습니다 — 위에 적은 재귀 손상이
실제로 일어나지 않는지 확인하는 것이 이 게이트의 목적이었습니다. `test_archive` 78/78.

## 부록 E. mmap 제로카피 (적용, Linux 기본 ON)

3번의 플랫폼 차이 때문에 `JDOC_USE_MMAP`은 Linux에서만 기본 ON입니다(그 외 OFF).
매핑이 실패하면 어느 빌드든 read 경로로 자동 폴백합니다.

### 설계: 맵은 뷰 전용

첫 구현은 `read_at`도 매핑에서 memcpy하도록 했는데, **3.3GB deflate 아카이브의 maxRSS가
152MB → 3.2GB로 폭증**했습니다. 압축 아카이브는 전 내용을 `read_at`으로 읽는데 그 바이트가
전부 매핑에 상주하게 되고, 정작 디코더는 어차피 사본이 필요하므로 이득이 0입니다.

그래서 **매핑은 `view_at`(제로카피 뷰)에만 쓰고 `read_at`은 항상 `pread`** 하도록 했습니다.
이러면 페이지가 실제로 복사를 아끼는 곳에만 쌓입니다. 결과적으로 압축 아카이브는 시간·RSS가
read 경로와 동일해지고, store 아카이브만 이득을 봅니다.

추가로 `view_at`이 누적 256MiB를 넘기면 `MADV_DONTNEED`로 상주분을 떨굽니다. 읽기 전용
`MAP_PRIVATE` 매핑이라 데이터 손실 없이 다음 접근에서 파일에서 다시 읽힙니다.

### 실측 (정본 머신, 교차 A/B 9회 중앙값)

| 픽스처 | read | mmap | 시간 | maxRSS |
|---|---|---|---|---|
| store 641MB | 0.396s | 0.073s | **−81.6%** | 151.3 → **27.0 MB (−82%)** |
| deflate 641MB | 1.839s | 1.855s | +0.9% (노이즈) | — |
| deflate 3.3GB | 9.176s | 9.172s | ±0% | 152.5 → 152.3 MB (동일) |
| 5중 중첩 258MB | 0.685s | 0.665s | −2.9% | — |

store 워크로드가 **5.4배** 빨라지면서 피크 RSS도 82% 줄었습니다. 압축 아카이브에는 아무
영향이 없으므로 손해 보는 구간이 없습니다.

### 부수 수정: 64비트 오프셋

기존 `read_at`은 `fseek(fp_, static_cast<long>(offset), SEEK_SET)`이었습니다. `long`이
32비트인 플랫폼(Windows LLP64, 32비트 리눅스)에서 **2GiB 이상 오프셋이 조용히 잘립니다** —
zip64 멤버가 정확히 그 영역입니다. `pread`(`off_t`)로 바꾸면서 해소했고, 파일 위치를
건드리지 않으므로 `const` 의미와도 맞고 스레드 안전해졌습니다.

### 동작 변화와 회귀 테스트

파일 기반 store 멤버가 `CapSink` 경로에서 `precheck_view_member` 경로로 옮겨갑니다. 캡 위반
시 보고되는 `uncompressed_size`가 "실제로 디코드된 부분 길이"에서 "헤더 선언 크기"로 바뀌는데,
store는 둘이 같으므로 관측 가능한 차이는 없습니다. 이를 계약으로 고정하려고
`file_backed_store_member_cap` / `file_backed_store_member_total_cap` 두 테스트를 추가했고,
`JDOC_USE_MMAP` ON/OFF 양쪽에서 동일하게 통과합니다(각 80/80).

### 리스크

매핑된 파일이 순회 중 truncate되면 사라진 페이지 접근에서 **SIGBUS**가 납니다. 라이브러리가
`sigaction`을 설치하는 것은 부적절하므로, 동시에 쓰이는 파일을 다루는 호출자는
`-DJDOC_USE_MMAP=OFF`로 빌드해야 합니다. 헤더 주석에 명시했습니다.
